| 権限者 | 複数可 | 部長が追加・削除 |
| 一般サークル員 | 複数可 | 部長が追加・削除 |

### 大方針K詳細: 特別イベント集計フォーム

- 部長のみがフォームを作成・削除できる
- 部長が任意のメンバー（一般員含む）にフォーム内限定の権限者を付与可能
- フォーム内権限者はそのフォーム内で支払い承認・催促通知等を行える
- フォーム内権限はそのイベント限り、グループ全体の権限には影響しない
- 月次リセットとは独立し、イベント単位で完結する

### 大方針M詳細: ユーザー認証・グループ参加制御

**アカウント登録:**
- メールアドレス等で個人が新規登録（1人1アカウント）
- 認証済みユーザーのみアプリ利用可

**グループ参加フロー:**
1. 部長がアプリ内で「招待リンク生成」（グループパスワード設定済み）
2. 部長が招待リンクをコピーし、LINE等で対象者に共有
3. 受け取った人がリンクを開く → パスワード入力画面
4. パスワード正解 → 部長に参加リクエストが届く
5. 部長が承認 → グループに一般員として追加

**招待リンク仕様:**
- 有効期限: 7日間
- グループパスワード: 部長がいつでも変更可能

### 大方針N詳細: 個人情報保護・データセキュリティ

**セキュリティ対策:**

| # | 対策 | 内容 |
|---|------|------|
| 1 | 通信の暗号化 | HTTPS必須（Vercel/Supabase標準対応） |
| 2 | DB行レベルアクセス制御 | RLSで自グループのデータのみアクセス可能 |
| 3 | 認証トークン管理 | JWTの有効期限設定、リフレッシュトークンのセキュア管理 |
| 4 | パスワードのハッシュ化 | グループパスワードはbcrypt等でハッシュ保存（平文保存禁止） |
| 5 | 入力値バリデーション | XSS・SQLインジェクション対策（サーバー側で必ず検証） |
| 6 | 個人情報の最小収集 | 必要最低限の情報のみ収集（表示名+メールアドレス） |
| 7 | メールアドレスの非公開 | 全ロールで他メンバーのメールアドレス閲覧不可。表示名のみ公開 |
| 8 | アカウント削除権 | ユーザーが自分のアカウントと関連データを削除可能 |

**権限マトリクス追加:**

| 操作 | 対応方針 | 部長 | 権限者 | 一般員 |
|------|---------|:----:|:------:|:-----:|
| 他メンバーのメールアドレス閲覧 | N | - | - | - |
| 自分のアカウント削除 | N | o | o | o |

### 大方針O詳細: イベント投票機能

**概要:** 権限者以上がイベント作成・編集時に、テキスト選択肢付きの投票を任意で追加できる。メンバーが投票した結果は、イベント詳細画面で名前横にバッジ表示される。

**投票仕様:**

| 項目 | 内容 |
|------|------|
| 投票数 | 1イベントにつき1つ（任意） |
| 質問文 | 必須（1〜50文字） |
| 選択肢 | 2〜10個、各1〜20文字 |
| 投票権 | 全メンバー（1人1票、変更可、投票しなくてもOK） |
| 表示 | イベント詳細のメンバー名横にバッジ表示（例: `佐藤 [月曜]`） |
| 集計 | 各選択肢の投票数を表示 |

**DB設計:**

| テーブル | カラム | 概要 |
|---------|--------|------|
| event_polls | id, event_id(unique), question(text) | イベントに紐づく投票（1:1） |
| event_poll_options | id, poll_id(FK), label(text), sort_order(int) | 選択肢（2〜10件） |
| event_poll_votes | poll_id(FK), user_id(FK), option_id(FK), unique(poll_id,user_id) | メンバーの投票（1人1票） |

**UI変更箇所:**

| 画面 | 変更内容 |
|------|---------|
| イベント作成 (events/new) | 投票セクション追加（質問文＋選択肢入力） |
| イベント編集 (EditEventForm) | 投票の追加・編集・削除 |
| イベント詳細 (events/[eventId]) | 投票UI＋メンバー名横にバッジ表示＋集計 |

### 大方針P詳細: メール送信基盤（Resend）

**背景:** Supabase Auth内蔵メール（無料枠）はレート制限が厳しく（1時間に3〜4通）、確認メール・パスワードリセットの開発デバッグに支障。

**方針:**
- Resend（resend.com）をメール送信サービスとして採用
- SupabaseのカスタムSMTP設定にResendのSMTP情報を入力
- コード変更なし（Supabaseダッシュボード設定のみ）

**Resend仕様:**

| 項目 | 内容 |
|------|------|
| 無料枠 | 100通/日 |
| 時間制限 | なし（秒間1通） |
| 設定方法 | Supabase Settings > Auth > SMTP Settings |
| 送信対象 | 確認メール、パスワードリセットメール |

**作業手順:**
1. Resendアカウント作成
2. 送信ドメイン設定（または共有ドメインで開発利用）
3. SMTP認証情報取得
4. Supabaseダッシュボードに入力
5. 送信テスト

### 中1 技術スタック（確定: 2026-05-20）

| 項目 | 選定 |
|------|------|
| フロントエンド | Next.js (React) |
| バックエンド | Next.js API Routes + Supabase |
| データベース | PostgreSQL (Supabase) |
| 認証 | Supabase Auth |
| リアルタイム | Supabase Realtime |
| ホスティング | Vercel (フロント) + Supabase (DB/Auth) |
| 言語 | TypeScript |

**決定理由:** Supabaseに認証・DB・リアルタイムを一元化し開発・運用コスト最小化。Vercel+Supabase無料枠で初期費用ゼロ運用可能。

### 中2 認証方式（確定: 2026-05-20）

- **Supabase Auth** を採用（中1の技術スタック選定により自動確定）
- メールアドレス+パスワードによるログイン/サインアップ
- SNSログイン（Google等）は将来的に追加検討
- RLS（Row Level Security）と連携し、DBレベルでのアクセス制御を実現

### 中3 リアルタイム通信方式（確定: 2026-05-21）

**方式:** Supabase Realtimeを使い、DBの変更をクライアント（ブラウザ/PWA）に自動プッシュする。

**リアルタイム反映対象:**

| 画面/機能 | 対象データ |
|-----------|-----------|
| 支払い承認画面 | 支払いステータスの変更 |
| 会計ダッシュボード | 集金状況の更新 |
| ホーム（掲示板） | 新規イベント・通知 |
| 特別イベント詳細 | 支払い状況の更新 |

**リアルタイム不要（通常API取得）:** メンバー一覧、権限者管理、招待リンク管理、ログイン/新規登録

**排他制御:** 楽観的ロック（versionカラム）を採用。同時書き込み時はバージョン不一致で弾き、クライアントに再取得を促す。

排他制御が必要な操作: 支払い承認、支払額調整、部長委譲、メンバー削除

**個人情報保護（大方針N対応・2026-05-21追記）:** Realtimeで配信するデータにメールアドレス等の個人情報を含めない。profilesテーブルの変更はsubscribe対象外とし、メンバー表示は display_name のみで行う。

**決定理由:** Supabase RealtimeはRLSと連携し自グループのデータ変更のみ受信可能。楽観的ロックは同時書き込みが少ないアプリに適し、DBロックで待たせない。

### 中4 通知方式（確定: 2026-05-21）

**通知チャネル:**

| 優先度 | チャネル | 用途 |
|--------|---------|------|
| 主 | Web Push（PWAプッシュ通知） | アプリを閉じていても届く。サービスワーカーで受信 |
| 常時 | アプリ内通知（通知一覧画面） | アプリ内で確認できる履歴。Realtime経由で自動表示 |

**通知トリガー:**

| トリガー | 通知先 | 内容 |
|----------|--------|------|
| 権限者が催促通知を送信 | 対象の未払い者 | 「○○の支払いが未完了です」 |
| 新規イベント作成 | グループ全員 | 「新しい集金項目が掲示されました」 |
| 支払い申告 | 権限者全員 | 「○○さんが支払い申告しました」 |
| 参加リクエスト | 部長 | 「○○さんが参加を希望しています」 |
| 部長委譲 | 新部長 | 「部長に任命されました」 |

**技術構成:** DBに通知レコード挿入 → Edge Functionが検知 → Web Push API送信。

**決定理由:** Web Pushはストア配布不要のPWAでスマホ通知を実現する最適手段。アプリ内通知で履歴を保持。

### 中8 セキュリティ実装方針（確定: 2026-05-21）

**N-1 通信暗号化:** HTTPS必須（Vercel/Supabaseデフォルト対応）。HSTSヘッダー設定。

**N-2 RLS（行レベルアクセス制御）:**

| テーブル | RLSポリシー |
|---------|------------|
| profiles | 本人は全列SELECT可。同グループメンバーはdisplay_nameのみ取得可（emailはテーブルに保持しない） |
| groups | 所属メンバーのみ全操作可能 |
| memberships | 同グループSELECT可、INSERT/UPDATE/DELETEは部長のみ |
| events, payment_statuses 等 | 同グループメンバーのみ（role別に操作制限） |
| event_polls, event_poll_options | 同グループSELECT全員、INSERT/UPDATE/DELETEは権限者以上 |
| event_poll_votes | 同グループSELECT全員、INSERT/UPDATE/DELETEは本人のみ |
| push_subscriptions | 本人のみ全操作可能 |

**N-3 認証トークン管理:** アクセストークン有効期限1時間（Supabaseデフォルト）。リフレッシュトークンはHttpOnly Cookieで保存（JSアクセス不可）。自動更新。

**N-4 パスワードハッシュ化:** ユーザーパスワードはSupabase Auth内部bcrypt。グループパスワードはサーバー側でbcryptハッシュ化して保存。

**N-5 入力値バリデーション:** フロントエンド: zodで型バリデーション（UX目的、信頼しない）。API Routes: zodで再バリデーション（これを信頼する）。DB: RLS+CHECK制約で最終防衛。XSS: Reactが標準エスケープ、dangerouslySetInnerHTML使用禁止。

**N-6 個人情報の最小収集:** 表示名+メールアドレスのみ。電話番号・住所・本名等は一切収集しない。

**N-7 メールアドレス非公開:** profilesテーブルにemailを保持しない（auth.getUser()経由で本人のみ取得可能）。Realtimeのsubscribe対象からprofilesを除外。

**N-8 アカウント削除フロー:** ユーザー要求 → 確認ダイアログ → 関連テーブルのuser_idを匿名IDに置換（集計保持） → profiles削除 → Supabase Authアカウント削除 → push_subscriptions削除 → ログアウト。部長は先に委譲してからのみ削除可能。

**決定理由:** 個人情報を扱うアプリとして最低限必要な保護を多層防御で実現。Supabaseの機能（RLS, Auth, HTTPS）を最大活用し、自前実装を最小化。

| Phase 2 | コア集金機能 | イベント作成→申告→承認フローが動く | 2週間 |
| Phase 3 | 管理・ダッシュボード | 管理画面・会計ダッシュボード | 1.5週間 |

**Phase 2 — コア集金機能:** イベントCRUD、支払い申告・承認、サブステータス・支払額調整、下部タブナビ構築。画面: #5,#7,#10,#11

**Phase 3 — 管理・ダッシュボード:** 会計ダッシュボード、メンバー管理・権限者管理・部長委譲、メンバー一覧。画面: #8,#12,#13,#14

**Phase 4 — 通知・リアルタイム:** アプリ内通知一覧、Web Push（SW導入）、メール通知（Resend）、Realtime導入。画面: #6

**Phase 5 — 特別イベント・仕上げ:** 特別イベントフォーム、アカウント設定・削除、PWA最適化、セキュリティ監査。画面: #9,#15,#16,#17

**依存関係:** Phase 1→2→3 は順序依存。Phase 4 は Phase 2 完了後に着手可能（Phase 3 と並行可）。Phase 5 は最後。

**決定理由:** 認証・DB基盤を最初に固め、コア機能（集金フロー）を早期に動作確認できる順序。通知・リアルタイムはコア完成後に追加する方がデバッグしやすい。

## 小方針

### Phase 1 小方針（確定: 2026-05-21）

**小1-1 Next.jsプロジェクト構成:**
- ルーター: App Router（Next.js 14+標準）
- ディレクトリ: src/app/ 配下にURL構造対応フォルダ
- スタイリング: Tailwind CSS
- UIコンポーネント: shadcn/ui
- フォーム: React Hook Form + zod
- 状態管理: React標準（useState/useContext）、グローバル状態管理ライブラリ不要
- パッケージマネージャ: npm

**小1-2 Supabase初期セットアップ:**
- プラン: 無料プラン（1プロジェクト）
- リージョン: Northeast Asia（東京）
- 初期テーブル: profiles, groups, memberships, invite_links, join_requests
- RLS: Phase 1対象テーブル全てに即時設定
- マイグレーション: Supabase CLI（supabase migration）でSQL管理

**小1-3 認証フロー詳細:**
- 新規登録入力: メールアドレス + パスワード + 表示名
- パスワード要件: 8文字以上（Supabase Authデフォルト）
- ログイン方式: メール+パスワードのみ（Phase 1時点）
- 認証後リダイレクト: /groups
- 未認証アクセス: ミドルウェアで /login にリダイレクト
- profiles作成: Auth登録時にDBトリガーで自動作成

**小1-4 グループ作成フロー詳細:**
- 入力: グループ名 + グループパスワード
- グループ名制限: 1〜30文字
- パスワード制限: 4〜20文字（英数字）
- 作成後: 作成者を部長としてmemberships追加 → グループホームへ遷移
- 複数グループ所属: 可能（/groups で切替）

**小1-5 招待リンク参加フロー詳細:**
- リンク形式: https://{domain}/invite/{ランダムtoken}
- トークン: UUIDv4（推測不可能）
- 有効期限: 7日間
- 参加フロー: リンクアクセス → 未認証なら /login → 認証後パスワード入力 → 部長に通知 → 部長承認 → 一般員追加
- 期限切れ: エラーメッセージ表示
- 重複参加防止: 既に所属済みならエラー表示

### Phase 2 小方針（確定: 2026-05-21）

**小2-1 イベントデータ仕様:**
- title: text, 必須, 1〜50文字
- amount: integer, 必須, 1〜999,999円
- due_date: date, 任意（支払い期限）
- description: text, 任意, 0〜200文字
- month: text, 必須（例: "2026-05"）、月次リセットのキー
- created_by: uuid, 必須（作成者ID）

**小2-2 支払いステータス状態遷移:**
- unpaid（未払い）→ claimed（申告中）→ paid（支払い済み）
- claimed → unpaid（差戻し）も可能
- イベント作成時、グループ全メンバー分のpayment_statusesをunpaidで一括生成
- 一般員: 自分のステータスのみ unpaid→claimed
- 権限者以上: claimed→paid（承認）または claimed→unpaid（差戻し）

**小2-3 月次リセット仕様:**
- 自動リセットはしない。月ごとに新しいイベントを作成する運用
- 過去月は閲覧可能（ホームで月フィルタ切替）
- デフォルト表示: 当月のイベントのみ

**小2-4 サブステータス仕様:**
- 入力形式: 自由テキスト（0〜50文字）
- 権限者以上のみ閲覧・編集
- 用途: 内部メモ（「銀行振込予定」「来週払うと連絡あり」等）

**小2-5 支払額調整仕様:**
- 調整可能条件: 2ヶ月連続未払い（前月・前々月にunpaidが残存）
- 調整範囲: 0〜999,999円（adjusted_amountに設定）
- 表示: 調整済みメンバーは支払い承認画面で金額横にマーク表示

**小2-6 下部タブナビゲーション実装:**
- 4タブ: ホーム / 通知 / メンバー / 設定
- 通知バッジ: Phase 4で実装、Phase 2では空
- レイアウトコンポーネント（src/app/g/[groupId]/layout.tsx）に配置

### Phase 3 小方針（確定: 2026-05-21）

**小3-1 会計ダッシュボード表示仕様:**
- 権限者以上: 集金進捗（X/Y人、プログレスバー）、集金額（現在/予定総額）、個別状況一覧（メンバー名・ステータス・サブステータス・調整額）、月切替
- 一般員: 集金進捗（X/Y人、プログレスバー）、自分の状態のみ。他メンバーの個別状況・金額詳細は閲覧不可

**小3-2 メンバー管理画面（部長のみ）:**
- 表示: 表示名・ロール・参加日
- 削除: 確認ダイアログ後に削除、関連payment_statusesも削除
- 制約: 部長自身は削除不可（先に委譲が必要）

**小3-3 権限者管理・部長委譲画面（部長のみ）:**
- 権限者追加: メンバー一覧から選択して一般員→権限者に昇格
- 権限者削除: 権限者→一般員に降格
- 部長委譲: メンバー選択 → 確認ダイアログ（「部長権限を失います」）→ 委譲実行
- 委譲後: 旧部長は一般員に降格、新部長に通知

**小3-4 メンバー一覧画面（全員）:**
- 表示: 表示名・ロール（バッジ表示）
- 非表示: メールアドレス（大方針N対応）
- ソート: 部長→権限者→一般員の順

### Phase 4 小方針（確定: 2026-05-21）

**小4-1 アプリ内通知一覧仕様:**
- 表示: 通知種別アイコン + メッセージ + 時刻
- 未読管理: is_readフラグ、一覧を開いた時点で既読に更新
- タブバッジ: 未読件数を数字バッジ表示
- 保持期間: 90日（古い通知はDBから定期削除）
- ページネーション: 20件ずつ無限スクロール

**小4-2 Web Push実装仕様:**
- サービスワーカー: public/sw.js に配置、Next.js PWA設定と統合
- 購読フロー: 初回ログイン後に通知許可ダイアログ → 許可時 push_subscriptions に登録
- 未許可時: アプリ内通知のみ（再度の許可促しはしない）
- 送信: Supabase Edge Function が DB変更検知 → Web Push API呼び出し
- VAPID鍵: Edge Functionのシークレットに保存

**小4-3（廃止）:** メール通知は採用しない。Web Push未許可ユーザーはアプリ内通知のみ。

**小4-4 Supabase Realtime導入仕様:**
- subscribe対象: payment_statuses, events, notifications（group_idフィルタ）
- subscribe除外: profiles（大方針N対応）
- クライアント側: カスタムフック useRealtimeSubscription(tableName, groupId) を作成
- 接続管理: グループホームのlayout.tsxでsubscribe、離脱時にunsubscribe
- エラー時: 自動再接続（Supabase SDKデフォルト）

### Phase 5 小方針（確定: 2026-05-21）

**小5-1 特別イベントフォーム仕様:**
- 作成入力: タイトル(1〜50文字) + 金額 + 説明(任意, 0〜200文字)
- フォーム内権限者: 部長がメンバーから選択して付与、複数名可
- 権限範囲: そのフォーム内のみ（支払い承認・催促通知・サブステータス）
- 支払いステータス: 通常イベントと同じ状態遷移（unpaid→claimed→paid）
- 月次リセットとは独立、イベント単位で完結、手動削除するまで残る
- 削除: 部長のみ、確認ダイアログ後に関連データごと削除

**小5-2 アカウント設定画面仕様:**
- 表示名変更: 現在の表示名を編集・保存（1〜20文字）
- メールアドレス表示: 自分のみ閲覧可（変更不可、確認用）
- アカウント削除: 確認ダイアログ2段階（1.確認メッセージ 2.パスワード再入力）→ 削除実行
- 部長の削除制約: 部長ロールのグループがある場合は「先に部長を委譲してください」と表示し削除不可

**小5-3 PWA最適化仕様:**
- マニフェスト: public/manifest.json にアプリ名・アイコン・テーマカラー・start_url
- アイコン: 192x192, 512x512 の2サイズ（PNG）
- display: standalone（ブラウザUI非表示）
- オフライン対応: SWでシェル（HTML/CSS/JS）をキャッシュ、データはオンライン必須
- インストール促進: ホーム画面に「インストール」バナー表示（A2HSイベント利用）

**小5-4 セキュリティ監査チェックリスト:**
1. 全テーブルにRLSが有効か
2. profiles.emailが本人以外から取得不可か
3. グループパスワードがハッシュ保存か
4. 全APIでzodバリデーション実装済みか
5. dangerouslySetInnerHTML未使用か
6. 未認証アクセス可能ページが /login, /signup, /invite/[token] のみか
7. 排他制御（version）が対象テーブルで機能しているか
8. アカウント削除後にデータが適切に匿名化されているか

## 決定理由

- PWA採用: ストア審査不要で他サークルへの展開が容易
- グループ概念: マルチサークル対応でデータ混同を防止
- 3階層ロール: 部長のみ委譲制で情報管理の一貫性を担保
- 特別イベントフォーム内権限: 合宿等の一時的な業務委任を柔軟に実現
- 限定リンク+パスワード二段階: リンク漏洩・パスワード漏洩の単独では参加不可、安全性向上
