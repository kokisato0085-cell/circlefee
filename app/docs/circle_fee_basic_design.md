# サークル費用管理アプリ 基本設計書

最終更新: 2026-05-27
ステータス: 大方針確定（A〜T） / 中方針全確定（中1〜中8） / 小方針全確定（Phase 1〜5）

---

## 1. プロジェクト概要

大学サークルの会費・集金を管理するWebアプリケーション。
複数サークルが独立して利用でき、将来的に他サークルへの展開を想定する。

### 1.1 コンセプト

- サークルの会費徴収・管理を効率化する
- 権限管理により情報の安全性を確保する
- スマートフォン（Android/iOS）から手軽にアクセスできる

---

## 2. システム構成

### 2.1 提供形態

| 項目 | 内容 |
|------|------|
| アプリ形態 | PWA（Progressive Web App） |
| 対応端末 | Android / iOS（ブラウザ経由） |
| データ保管 | サーバー上に永続化 |
| 同時アクセス | 複数ユーザーが同時操作可能 |
| リアルタイム通信 | Supabase Realtimeによる自動プッシュ |
| 排他制御 | 楽観的ロック（versionカラム） |

### 2.2 システム構成図（概要）

```
[ブラウザ/PWA]  ←subscribe→  [Supabase Realtime]  ←→  [PostgreSQL]
  (複数端末)    ←API→        [Next.js API Routes]  ←→  [Supabase]
```

### 2.3 リアルタイム反映対象

| 画面/機能 | 対象データ | リアルタイム |
|-----------|-----------|:-----------:|
| 支払い承認画面 | 支払いステータスの変更 | o |
| 会計ダッシュボード | 集金状況の更新 | o |
| ホーム（掲示板） | 新規イベント・通知 | o |
| 特別イベント詳細 | 支払い状況の更新 | o |
| メンバー一覧 | メンバー情報 | - |
| 権限者管理 | 権限情報 | - |
| 招待リンク管理 | リンク情報 | - |

### 2.4 排他制御

楽観的ロック方式。各レコードに`version`カラムを持たせ、更新時にバージョン一致を確認。不一致時はクライアントに再取得を促す。

対象操作: 支払い承認、支払額調整、部長委譲、メンバー削除

### 2.5 リアルタイム配信時の個人情報保護（大方針N対応）

Realtimeで配信するデータにメールアドレス等の個人情報を含めない。profilesテーブルの変更はsubscribe対象外とし、メンバー表示は display_name のみで行う。

## 3. ユーザーモデル

### 3.1 グループ（サークル単位）

- アプリ上の最上位単位
- グループ間のデータは完全に分離
- グループ発足人が初代部長に自動就任

### 3.2 ロール定義

| ロール | 人数制限 | 就任方法 | 備考 |
|--------|---------|----------|------|
| 部長 | グループに1人 | 発足時自動就任 / 委譲 | 委譲したら元の人は権限を失う |
| 権限者 | 複数可 | 部長が追加・削除 | — |
| 一般サークル員 | 複数可 | 部長が追加・削除 | — |

権限の強さ: **部長 > 権限者 > 一般サークル員**
部長は権限者の全操作を包含する（上位互換）。

### 3.3 権限マトリクス

| 操作カテゴリ | 操作 | 部長 | 権限者 | 一般員 |
|-------------|------|:----:|:------:|:-----:|
| **ロール管理** | 部長を委譲 | o | - | - |
| | 権限者の追加・削除 | o | - | - |
| **メンバー管理** | メンバーの追加・削除 | o | - | - |
| | グループ名変更 | o | - | - |
| **集金管理** | イベント（集金項目）作成・掲示 | o | o | - |
| | 支払い承認（済にする） | o | o | - |
| | サブステータス閲覧・付与 | o | o | - |
| | 未払い者の支払額調整 | o | o | - |
| | 催促通知の送信 | o | o | - |
| **特別イベント** | フォーム作成・削除 | o | - | - |
| | フォーム内権限者の付与 | o | - | - |
| **閲覧（権限者以上）** | 会計ダッシュボード（詳細） | o | o | - |
| **バッジ管理** | 学年バッジ付与・変更 | o | - | - |
| | 係バッジの作成・削除 | o | - | - |
| | 係バッジの付与・解除 | o | - | - |
| **招待管理** | 招待リンク生成 | o | - | - |
| | グループパスワード変更 | o | - | - |
| | 参加リクエスト承認 | o | - | - |
| **閲覧（全員）** | イベント・掲示の閲覧 | o | o | o |
| | メンバー一覧の閲覧 | o | o | o |
| | 集金状況（簡易: X/Y人済） | o | o | o |
| **申告** | 「支払った」申告 | o | o | o |
| | 自分のステータス確認 | o | o | o |
| **セキュリティ** | 他メンバーのメールアドレス閲覧 | - | - | - |
| | 自分のアカウント削除 | o | o | o |

## 4. 機能概要

### 4.1 グループ管理（大方針J）

- グループの作成（発足人=初代部長）
- グループ名の変更（部長のみ）
- メンバーの追加・削除（部長のみ）
- グループ間のデータ完全分離

### 4.2 イベント（集金項目）管理（大方針D）

- 権限者以上がイベントを作成
- イベントに含まれる情報: 支払日・場所・額 等
- 作成したイベントを全員に掲示

### 4.3 支払いステータス管理（大方針E）

- 一般員が「支払った」を申告
- 権限者以上が申告を確認し承認 → ステータスが「支払い済み」に変更
- ステータスは月単位でリセット

### 4.4 サブステータス（大方針F）

- 権限者以上のみが閲覧・付与できる補足ステータス
- 一般員からは見えない

### 4.5 未払い者への支払額調整（大方針G）

- 2ヶ月連続未払いの場合、権限者以上が支払額を増減可能

### 4.6 催促通知・通知システム（大方針C / 中4確定）

**通知チャネル:**

| 優先度 | チャネル | 備考 |
|--------|---------|------|
| 主 | Web Push（PWAプッシュ通知） | サービスワーカーで受信、アプリ閉じても届く |
| 常時 | アプリ内通知一覧 | Realtime経由で自動表示、履歴として保持 |

※ メール通知は小4-3で廃止。Web Push未許可ユーザーはアプリ内通知のみ。

**通知トリガー:**

| トリガー | 通知先 |
|----------|--------|
| 権限者が催促通知を送信 | 対象の未払い者 |
| 新規イベント作成 | グループ全員 |
| 支払い申告 | 権限者全員 |
| 参加リクエスト | 部長 |
| 部長委譲 | 新部長 |

**処理フロー:** DBに通知レコード挿入 → Edge Functionが検知 → Web Push API送信

### 4.7 会計ダッシュボード（大方針H）

| 閲覧者 | 表示内容 |
|--------|---------|
| 権限者以上 | 支払い済み人数・金額、予定総額、個別状況など詳細 |
| 一般員 | 今月 X/Y人 支払い済み（簡易表示のみ） |

- 支払い済みステータスの数から現在の集金額を算出
- 全員支払い時の予定総額を表示

### 4.8 特別イベント集計フォーム（大方針K）

- 部長のみがフォームを作成・削除
- 合宿等の通常月額とは別枠の集金を管理
- 部長が任意のメンバーにフォーム内限定の権限者を付与可能
- フォーム内権限者: そのフォーム内で支払い承認・催促通知等が可能
- フォーム内権限はそのイベント限り（グループ全体の権限には影響しない）
- 月次リセットとは独立、イベント単位で完結

### 4.9 メンバーバッジ（大方針S）

- 部長がメンバーに学年バッジ（1〜4年）を設定・解除
- 部長がグループごとに係バッジ（自由名称、1〜20文字）を作成・削除
- 部長が係バッジをメンバーに付与・解除（1メンバーに複数可）
- メンバー一覧・イベント詳細・支払い管理・承認画面でバッジを表示

### 4.10 ユーザー認証・グループ参加制御（大方針M）

**アカウント登録:**
- メールアドレス等で個人が新規登録（1人1アカウント）
- 認証済みユーザーのみアプリ利用可

**グループ参加フロー:**
1. 部長がアプリ内で「招待リンク生成」（グループパスワード設定済み）
2. 部長が招待リンクをコピーし、LINE等で対象者に共有
3. 受け取った人がリンクを開く → パスワード入力画面
4. パスワード正解 → 部長に参加リクエストが届く
5. 部長が承認 → グループに一般員として追加

**招待リンク仕様:**

| 項目 | 内容 |
|------|------|
| 有効期限 | 7日間 |
| パスワード変更 | 部長がいつでも変更可能 |
| 共有手段 | アプリ内でリンクをコピー → LINE等で共有 |

## 5. データモデル（中5確定）

### 5.1 テーブル一覧

| テーブル名 | 概要 | 主要カラム |
|-----------|------|-----------|
| profiles | ユーザー情報 | id(=Auth UID), display_name |
| groups | グループ | id, name, password_hash, created_by |
| memberships | 所属・ロール | group_id, user_id, role, version, grade |
| invite_links | 招待リンク | group_id, token, expires_at |
| join_requests | 参加リクエスト | group_id, user_id, status |
| events | 集金イベント（月次） | group_id, title, amount, due_date, month |
| payment_statuses | 支払いステータス | event_id, user_id, status, sub_status, adjusted_amount, version, claim_date, claim_place, claim_recipient, claim_message |
| special_events | 特別イベント | group_id, title, amount, created_by |
| special_event_roles | フォーム内権限者 | special_event_id, user_id |
| special_payment_statuses | 特別イベント支払い | special_event_id, user_id, status, version |
| notifications | 通知 | group_id, target_user_id, type, message, is_read |
| group_roles | 係バッジ定義 | id, group_id, name |
| member_roles | 係の割り当て | group_role_id, membership_id |
| push_subscriptions | Web Push購読情報 | user_id, endpoint, keys_json |

### 5.2 リレーション

```
profiles ──1:N── memberships ──N:1── groups
groups   ──1:N── events      ──1:N── payment_statuses
groups   ──1:N── special_events ──1:N── special_payment_statuses
                                  └─1:N── special_event_roles
groups   ──1:N── invite_links
groups   ──1:N── join_requests
groups   ──1:N── group_roles ──1:N── member_roles ──N:1── memberships
profiles ──1:N── notifications
profiles ──1:N── push_subscriptions
```

### 5.3 設計ポイント

- profiles.id = Supabase Auth の auth.users.id で紐付け
- 排他制御: memberships, payment_statuses, special_payment_statuses に version カラム
- RLS: 全テーブルに group_id ベースの Row Level Security を適用
- 月次リセット: events に month カラムで月ごと管理

### 5.4 個人情報保護（大方針N対応）

- emailはprofilesテーブルに保存しない（auth.getUser()経由で本人のみ取得可能）
- アカウント削除時: profiles を削除し、関連テーブルの user_id 参照先を匿名化（"退会済みユーザー" に置換）。集計データの整合性を維持

## 9. セキュリティ（大方針N）

### 9.1 セキュリティ対策一覧

| # | 対策 | 内容 |
|---|------|------|
| 1 | 通信の暗号化 | HTTPS必須（Vercel/Supabase標準対応）、HSTSヘッダー設定 |
| 2 | DB行レベルアクセス制御 | RLSで自グループのデータのみアクセス可能 |
| 3 | 認証トークン管理 | アクセストークン1時間、リフレッシュトークンはHttpOnly Cookie |
| 4 | パスワードのハッシュ化 | ユーザー: Supabase Auth(bcrypt)、グループ: サーバー側bcrypt |
| 5 | 入力値バリデーション | フロント: zod(UX)、API: zod(信頼)、DB: RLS+CHECK制約 |
| 6 | 個人情報の最小収集 | 表示名+メールアドレスのみ。電話番号・住所・本名は不収集 |
| 7 | メールアドレスの非公開 | RLS+APIフィルタリング+Realtime除外の三重防御 |
| 8 | アカウント削除権 | 匿名化置換 → profiles削除 → Auth削除。部長は委譲後のみ |

### 9.2 RLSポリシー概要

| テーブル | ポリシー |
|---------|---------|
| profiles | 本人は全列SELECT可。同グループメンバーはdisplay_nameのみ取得可（emailはテーブルに保持しない） |
| groups | 所属メンバーのみ全操作可能 |
| memberships | 同グループSELECT可、INSERT/UPDATE/DELETEは部長のみ |
| events, payment_statuses 等 | 同グループメンバーのみ（role別に操作制限） |
| push_subscriptions | 本人のみ全操作可能 |

### 9.3 バリデーション方針

多層防御で実装。フロントエンドのバリデーションはUX目的で信頼しない。サーバー側（API Routes）のzodバリデーションを正とし、DB側のRLS+CHECK制約を最終防衛とする。ReactのXSS対策（標準エスケープ）を活用し、dangerouslySetInnerHTMLは使用禁止。

### 9.4 アカウント削除フロー

1. ユーザーが削除を要求 → 確認ダイアログ
2. 部長の場合: 先に部長を委譲しなければ削除不可
3. 関連テーブルの user_id を匿名IDに置換（集計データ保持）
4. profiles レコード削除
5. Supabase Auth アカウント削除
6. push_subscriptions 削除
7. ログアウト状態に遷移
